## Issues Found and Fixed During Testing

### 1. Missing `sabot-module.conf` (Critical)
**Symptom**: Dremio returned `"An invalid value was found: PINOT"` when adding source.
**Root cause**: Without `sabot-module.conf`, Dremio's classpath scanner does not scan the connector JAR for `@SourceType` annotations.
**Fix**: Added `src/main/resources/sabot-module.conf` containing:
```
dremio.classpath.scanning.packages += "com.dremio.exec.store.jdbc.conf"
```

### 2. Subquery pushdown rejected by Pinot single-stage broker
**Symptom**: `GROUP BY + ORDER BY + LIMIT` queries failed â€” Dremio generates subquery-wrapped SQL that Pinot's default broker rejects.
**Exact SQL generated by Dremio**:
```sql
SELECT "Carrier" AS "Carrier", "cnt"
FROM (SELECT "airlineStats"."Carrier", COUNT(*) AS "cnt"
      FROM "airlineStats" GROUP BY "airlineStats"."Carrier") AS "airlineStats"
ORDER BY "cnt" DESC LIMIT 10
```
**Fix**: Added `useMultiStageEngine=true` to the JDBC URL. Pinot's multi-stage query engine (Calcite-based) fully supports subqueries and this SQL form.

### 3. INT null-sentinel cast failures
**Symptom**: `SELECT *` on `airlineStats` threw `Failed to cast the string [-2147483648] to int32_t` â€” Pinot's `INT_ARRAY` columns return their null sentinel (`Integer.MIN_VALUE`) wrapped in array brackets.
**Fix A**: Mapped `INT`/`INTEGER` â†’ `bigint` in `pinot-arp.yaml` (Dremio's JDBC reader calls `getLong()` for bigint, handling the full int32 range).
**Fix B**: Added `enableNullHandling=true` to the JDBC URL so Pinot returns SQL NULL for null/sentinel values instead of the magic numbers.
**Fix C**: Patched `ResultTableResultSet.java` â€” when a JDBC cell is a JSON `ArrayNode` (multi-value column), extract the first element as a scalar string instead of the raw `toString()` which adds brackets.
