# Dremio ClickHouse Connector

*Built by Mark Shainman*

A Dremio storage plugin that adds **read support for ClickHouse databases**.

Uses Dremio's **ARP (Advanced Relational Pushdown)** framework — a declarative
YAML-driven approach where SQL operations (filters, aggregations, sorts, joins,
scalar functions) are pushed down to ClickHouse automatically without writing
custom planning rules. The official ClickHouse JDBC driver handles the wire
protocol (no Spark, no extra servers required).

---

## Architecture

```
SQL:  SELECT region, COUNT(*), AVG(amount)
      FROM clickhouse_source.analytics.events
      GROUP BY region ORDER BY COUNT(*) DESC

Dremio Planner (ARP)
  └── ArpDialect (clickhouse-arp.yaml)
        ├── Predicate pushdown  → WHERE col = ? pushed to ClickHouse SQL
        ├── Aggregation pushdown → GROUP BY, COUNT, SUM, AVG pushed down
        ├── Sort pushdown       → ORDER BY pushed down
        ├── Limit pushdown      → LIMIT / OFFSET pushed down
        └── Function mapping    → toDate, toYear, startsWith, … pushed down

ClickHouseConf (JdbcPluginConfig)
  └── ClickHouseDialect (ArpDialect)
        └── unparseDateTimeLiteral()  ← rewrites ANSI DATE/TIMESTAMP literals
                                         to toDate() / toDateTime()

JDBC → clickhouse-jdbc (HTTP interface, port 8123)
  └── ClickHouse server
```

### Key Classes

| Class | Role |
|---|---|
| `ClickHouseConf` | Source config shown in Dremio's "Add Source" UI. Extends `AbstractArpConf`. Defines connection fields (host, port, database, auth, SSL, pool, timeouts, performance). Builds JDBC driver properties for compression and server-side timeout. |
| `ClickHouseDialect` | Extends `ArpDialect`. Overrides `unparseDateTimeLiteral` to emit `toDate('…')` / `toDateTime('…')` instead of ANSI SQL `DATE '…'` / `TIMESTAMP '…'` syntax that ClickHouse rejects. |
| `clickhouse-arp.yaml` | The ARP dialect definition. Declares all supported types, pushdown capabilities, and scalar functions. Drives all SQL translation. |

### ARP vs. Native Driver Approach

| Aspect | ARP (this connector) | Native driver (Cassandra connector) |
|---|---|---|
| SQL translation | Declarative YAML | Custom Java planning rules |
| Predicate pushdown | Automatic via ARP | Manual filter rule classes |
| New function support | Add entry to YAML | Write new planning rule |
| Write support | Not available (ARP is read-only) | Can be implemented |
| Complex types (Array, Map) | VARCHAR fallback | Full native types possible |
| Development effort | Low | High |

ARP is the right choice here because ClickHouse speaks SQL and has a production-grade JDBC driver. ARP is **not** suitable for non-SQL sources (Cassandra, HBase) or sources that require custom write paths.

## Dremio 26.x Architectural Constraints

### ARP is read-only
The ARP framework (`AbstractArpConf` + `JdbcStoragePlugin`) exposes no write path. `CTAS`, `INSERT INTO`, `UPDATE`, and `DELETE` on ClickHouse sources are not supported. This is a framework constraint, not a ClickHouse limitation.

### DML is Iceberg-only
Even if write support were added, Dremio 26.x's `WriterCommitterOperator` only supports Iceberg-backed tables for DML operations.

## Performance Configuration

| Setting | Default | Notes |
|---|---|---|
| `ClickHouse Cloud` | false | One-click Cloud setup: forces port 8443 and SSL. Enable when connecting to ClickHouse Cloud (cloud.clickhouse.com). |
| `Port` | 8123 | HTTP interface. Use 8443 for HTTPS. Ignored when ClickHouse Cloud is on. |
| `Max Idle Connections` | 8 | JDBC connection pool size. Increase for high-concurrency workloads. |
| `Connection Timeout` | 30 s | Time to establish a TCP connection. |
| `Socket Timeout` | 300 s | Time to wait for query results. Increase for long-running analytical queries. |
| `Enable HTTP Result Compression` | true | LZ4 HTTP compression; reduces payload 3–10x for analytical result sets. Disable only if ClickHouse is co-located and bandwidth is unconstrained. |
| `Fetch Block Size (rows)` | 65 536 | Rows per response block from ClickHouse. Increase for large bulk scans; decrease for interactive / memory-constrained queries. |
| `Excluded Databases` | _(empty)_ | Comma-separated database names to hide from the Dremio catalog browser (e.g. `staging,raw_ingest`). System databases are always hidden. |
| `Additional JDBC Properties` | _(empty)_ | Extra `key=value` driver properties, one per line or semicolon-separated. Applied after built-in properties — can override defaults. Example: `session_timezone=UTC`. |

**Server-side execution timeout** is automatically set to `socketTimeout − 10 s` (minimum 10 s). This tells ClickHouse to cancel the query server-side before the client socket times out, ensuring clean query cancellation and no orphaned queries.

## Keeping Up with Dremio Upgrades

When Dremio is upgraded to a new version, run `rebuild.sh` to automatically detect the
new version, update `pom.xml`, rebuild the plugin JAR against the new Dremio JARs, and
redeploy — all in one command. The ClickHouse JDBC driver JAR (`clickhouse-jdbc-0.4.6-all.jar`)
is version-stable and does not need to change when Dremio upgrades.

```bash
./rebuild.sh                                    # Docker (default: try-dremio)
./rebuild.sh --docker my-dremio                 # Named Docker container
./rebuild.sh --local /opt/dremio                # Bare-metal
./rebuild.sh --k8s dremio-master-0              # Kubernetes (deploys both JARs to all pods)
./rebuild.sh --k8s dremio-master-0 -n dremio-ns # Kubernetes with namespace
./rebuild.sh --dry-run                          # Preview only, no changes
```

`rebuild.sh` also detects and updates `calcite.version` automatically, since Calcite
occasionally ships a new version alongside Dremio point releases.

## Known Limitations

### STDDEV / VAR — local fallback, no pushdown
Dremio's ARP framework generates SQL like `SUM(x * x) / COUNT(x)` for STDDEV, which embeds aggregate calls inside other aggregates. ClickHouse rejects this with `Code: 184. DB::Exception: Aggregate function … is found inside another aggregate function`. STDDEV, STDDEV_POP, VAR_SAMP, VAR_POP are excluded from the ARP pushdown list; Dremio computes them locally by pulling the raw column data. Results are correct — only performance is affected (no server-side aggregation).

### Array, Tuple, Map types — VARCHAR fallback
ClickHouse's complex types (`Array(T)`, `Tuple(...)`, `Map(K,V)`) are returned as
their string representations (`[1,2,3]`, `(a, b)`, `{'key': 'val'}`). Element-level
operations (e.g. `arrayElement(col, 1)`, `mapKeys(col)`) are not pushed down.

To work with ClickHouse array data, create a ClickHouse Materialized View that
flattens the arrays, then query the MV through Dremio.

### UInt128, UInt256, Int128, Int256
These types exceed the 64-bit range of Dremio's BIGINT. Values are returned as
VARCHAR string representations. Arithmetic operations on these columns are not
pushed down.

### Inline `VALUES` constant row in `UNION ALL`
When Dremio rewrites a `UNION ALL` branch containing a pure constant row (e.g. `SELECT 'baseline', 20.0`) it emits a JDBC `VALUES (…)` clause that ClickHouse's SQL dialect does not support.

**Workaround:** Source the constant from a real table with `LIMIT 1`:
```sql
-- Instead of:
SELECT 'baseline', 20.0
-- Use:
SELECT 'baseline', 20.0 FROM clickhouse_source.mydb.any_table LIMIT 1
```

## Building

```bash
# Build from source (produces plugin JAR)
mvn clean package -DskipTests

# The deployable JAR is:
target/dremio-clickhouse-connector-1.0.0-SNAPSHOT-plugin.jar
```

**Build note:** The Dremio JDBC plugin JAR (which provides `AbstractArpConf` and
the ARP framework) must be installed into your local Maven repo from the running
Dremio instance before building. The `install.sh --build` path does this automatically.
For manual builds, find the relevant JARs in `$DREMIO_HOME/jars/` and install them
with `mvn install:install-file`.

## Quick Start

### 1. Install the connector

```bash
# Docker (default container: try-dremio)
./install.sh --docker try-dremio --prebuilt

# Bare-metal
./install.sh --local /opt/dremio --prebuilt

# Kubernetes
./install.sh --k8s dremio-master-0
./install.sh --k8s dremio-master-0 --namespace dremio-ns
```

### 2. Restart Dremio

```bash
docker restart try-dremio          # Docker
systemctl restart dremio           # bare-metal
kubectl rollout restart statefulset/<name> -n <namespace>  # Kubernetes
```

### 3. Register a ClickHouse source

```bash
# Local / self-hosted (simplest)
./add-clickhouse-source.sh --name clickhouse --host localhost

# Named database + credentials
./add-clickhouse-source.sh \
  --name clickhouse_prod \
  --host clickhouse.example.com \
  --database analytics \
  --ch-user dremio_reader \
  --ch-pass mypassword

# ClickHouse Cloud (forces port 8443 + SSL automatically)
./add-clickhouse-source.sh \
  --name ch_cloud \
  --host abc123.us-east-1.aws.clickhouse.cloud \
  --cloud \
  --ch-user default \
  --ch-pass "$CH_CLOUD_PASSWORD"

# SSL + truststore for self-hosted HTTPS
./add-clickhouse-source.sh \
  --name clickhouse_ssl \
  --host clickhouse.example.com \
  --port 8443 --ssl \
  --ssl-truststore /certs/clickhouse-truststore.jks \
  --ssl-ts-pass truststorepass

# Preview the JSON payload without submitting
./add-clickhouse-source.sh --name clickhouse --host localhost --dry-run
```

Run `./add-clickhouse-source.sh --help` for the full list of options.

## Deployment

### Option A — Automated (recommended)

```bash
./install.sh                                    # guided interactive mode
./install.sh --docker try-dremio --prebuilt     # fast non-interactive deploy
./install.sh --docker try-dremio --build        # full build from source + deploy
```

The installer handles: build (optional), copy, deploy to `jars/3rdparty/`, restart, and health check.

### Option B — Manual

1. Copy both JARs to `$DREMIO_HOME/jars/3rdparty/`:
   ```
   dremio-clickhouse-connector-1.0.0-SNAPSHOT-plugin.jar
   clickhouse-jdbc-0.4.6-all.jar
   ```
2. Restart Dremio.
3. **Sources → Add Source → ClickHouse**
4. Fill in Host, Port (8123), Database, Username, Password.
5. Click **Save** — ClickHouse tables appear in the catalog.

## SQL Usage

```sql
-- Explore available tables
SHOW TABLES IN clickhouse_source.my_database;

-- Basic read — filters and projections pushed to ClickHouse
SELECT event_type, region, amount
FROM clickhouse_source.analytics.events
WHERE order_date >= TIMESTAMP '2025-01-01 00:00:00'
  AND region = 'us-east';

-- Aggregation pushed down entirely to ClickHouse
SELECT region, COUNT(*), AVG(amount), MAX(amount)
FROM clickhouse_source.analytics.events
GROUP BY region
ORDER BY COUNT(*) DESC;

-- Date functions pushed down
SELECT DATE_TRUNC('month', order_date) AS mo, SUM(amount) AS revenue
FROM clickhouse_source.sales.orders
GROUP BY 1
ORDER BY 1;

-- JOIN between ClickHouse tables
SELECT u.username, SUM(e.amount) AS total
FROM clickhouse_source.app.events e
JOIN clickhouse_source.app.users u ON e.user_id = u.id
GROUP BY u.username
ORDER BY total DESC
LIMIT 10;

-- Cross-source JOIN: ClickHouse + Hudi (Dremio federates automatically)
SELECT h.username, c.total_revenue
FROM hudi_local.sample_users h
JOIN clickhouse_source.analytics.revenue_by_user c ON h.user_id = c.user_id
ORDER BY c.total_revenue DESC;

-- String functions pushed down to ClickHouse
SELECT INITCAP(status), REGEXP_LIKE(email, '.*@company\\.com')
FROM clickhouse_source.mydb.orders
WHERE LOCATE('@', email) > 0;

-- TO_CHAR date formatting (Oracle-style format codes)
SELECT TO_CHAR(order_date, 'YYYY-MM-DD') AS date_str,
       TO_CHAR(order_date, 'MON YYYY') AS month_label,
       TO_CHAR(order_date, 'DY DD-MON-YY') AS short_label
FROM clickhouse_source.mydb.orders
LIMIT 5;

-- Group by formatted date
SELECT TO_CHAR(order_date, 'YYYY-MM') AS month, COUNT(*) AS orders, SUM(amount) AS revenue
FROM clickhouse_source.mydb.orders
GROUP BY 1 ORDER BY 1;
```

### Case-insensitive matching (ILIKE workaround)

Dremio 26.x's SQL parser does not support the `ILIKE` keyword. Use `REGEXP_LIKE` with
the `(?i)` flag instead — it pushes down to ClickHouse's `match()` and is equivalent:

```sql
-- Instead of: WHERE status ILIKE 'PEND%'
WHERE REGEXP_LIKE(status, '(?i)pend.*')

-- Instead of: WHERE email ILIKE '%@company.com'
WHERE REGEXP_LIKE(email, '(?i).*@company\\.com')
```

### Conditional expressions — use CASE WHEN

Dremio's SQL parser does not include `IIF`, `IF`, or `IFF` in its grammar. Use standard `CASE WHEN` instead — it pushes down to ClickHouse as-is:

```sql
-- Instead of: IIF(amount > 100, 'big', 'small')
SELECT CASE WHEN amount > 100 THEN 'big'
            WHEN amount > 50  THEN 'medium'
            ELSE 'small' END AS bucket,
       COUNT(*) AS n
FROM clickhouse_source.mydb.orders
GROUP BY 1;
```

### SPLIT_PART

```sql
-- Split a delimited string and return the Nth part (1-based)
SELECT SPLIT_PART(email, '@', 2) AS domain   -- 'user@example.com' → 'example.com'
FROM clickhouse_source.mydb.users;
```

### MEDIAN

```sql
-- Median of a numeric column — pushed to ClickHouse as median()
SELECT region, MEDIAN(amount) AS p50
FROM clickhouse_source.mydb.orders
GROUP BY region;
```

### What does NOT work

```sql
-- ❌ Writes not supported — ARP is read-only
INSERT INTO clickhouse_source.my_db.my_table SELECT ...;
CREATE TABLE clickhouse_source.my_db.new_table AS SELECT ...;

-- ⚠️ Array element access — not pushed down (column returned as VARCHAR)
SELECT col[1] FROM clickhouse_source.my_db.my_table;
-- Workaround: flatten in a ClickHouse Materialized View first
```

## Key Design Decisions

**Why ARP and not a native driver?**
ClickHouse speaks SQL and ships a production-grade JDBC driver. ARP lets us get
full predicate, aggregation, sort, and limit pushdown with a small amount of YAML
configuration — no custom Calcite planning rules needed. The Cassandra connector
required ~3,000 lines of custom planning code for equivalent pushdown; the ARP
connector achieves the same result declaratively.

**Why `clickhouse-jdbc` 0.4.6 and not a newer version?**
`clickhouse-jdbc-0.6.x-all.jar` from Maven Central no longer bundles `ClickHouseClient`
(moved to a separate Maven module in v0.6.0). The 0.4.6 `all` classifier JAR bundles
everything needed in a single file — no dependency management required in the Dremio
classpath. Version 0.4.6 supports ClickHouse 20.7+ including 24.x.

**Why remove the SPI entry from the driver JAR?**
Without this patch, Java's `ServiceLoader` auto-registers `ClickHouseDriver` during
JVM startup on a background thread. This causes a class-initialization race condition
inside Dremio's DBCP2 connection pool, permanently blocking all future connection
attempts with `NoClassDefFoundError`. The static pre-loader in `ClickHouseConf`
handles registration on the correct thread; the SPI entry is redundant and harmful.

**Why HTTP interface (port 8123) and not native TCP (port 9000)?**
The JDBC driver uses HTTP as its primary transport. HTTP is stateless, firewall-friendly,
and works well with LZ4 compression. For analytical queries the overhead vs. native TCP
is negligible.
